# Requirements

## 背景与目标
当前工作区提供了 OpenClaw 风格的代理运行环境，但用户需要的是一个面向本机 Codex 生态的“总经理控制台”：它既像 OpenClaw Control UI 一样有统一入口，又能围绕 Codex 的真实配置与执行能力完成项目总管、人格总管、记忆总管、知识库总管和 cron 总管。

目标不是再造一个新的聊天机器人，而是做一个“Codex 运营系统”：
- 让用户与一个全权总经理对话。
- 让总经理调度本机 Codex、项目级配置、角色模板、会话与长期自动化。
- 让用户也能通过前端直接管理这些对象，而不必所有动作都走聊天。

## 原始需求与约束
- 统一前端，作为总经理对话入口和全部管理入口。
- 总经理支持接入 Telegram / Slack；本机已有 OpenClaw Slack 能力可借鉴。
- 角色管理：
  - 人格、偏好、system prompt / project prompt、是否使用默认指令。
  - 工具集、MCP、技能、全局或项目作用域。
  - 外部聊天工作应用中的名字、说话规则、发言风格。
- 记忆管理：
  - 历史对话、归档、复用，不和角色强绑定。
- 知识库管理：
  - 使用本机 `6333` 端口上的 Qdrant。
- 项目管理：
  - 项目总经理角色、项目内 session、参与角色、项目级 MCP/技能/工具、知识库权限、外部频道关联。
- cron 定时任务管理：
  - 兼容现有 `cron-loop` 技能和项目中的 `.cron-loop/` 平铺目录。
- 约束：
  - 必须基于真实本机 Codex 能力设计。
  - 不破坏已有 `~/.codex` 和其他项目的 `.cron-loop` 结构。

## 纵向扩展需求
- Codex 运行桥接：
  - 支持 `codex exec`、`codex exec resume`、`codex mcp-server` 三条执行路径。
  - 支持模型、profile、reasoning、sandbox、approval policy、instructions、developer instructions 的覆盖。
- 角色物化机制：
  - 角色定义可生成 `.codex/config.toml` 片段、指令文件和运行参数。
  - 角色定义与项目定义组合后可形成“项目总经理运行快照”。
- 会话索引与归档：
  - 读取 `~/.codex/sessions`、`~/.codex/archived_sessions`、`history.jsonl`。
  - 支持标签、归属项目、归属角色、恢复入口。
- cron-loop 兼容层：
  - 扫描 `<project>/.cron-loop/*.state.json`、`*.latest.md`、`*.ledger.*` 等文件。
  - 支持 pause / resume / install / validate / log 查看。
- 知识库治理：
  - 管理 Qdrant collection 与项目映射。
  - 允许项目声明“可读知识库”和“可写知识库”。

## 横向扩展需求
- 统一审计与操作日志。
- WebSocket / SSE 事件流，用于总经理聊天过程与任务状态刷新。
- 外部渠道抽象层，便于 Slack / Telegram / 未来其他渠道接入。
- 仪表盘式概览：
  - 活跃项目、活跃会话、知识库、任务健康度、最近运行。
- 可导出的项目快照与配置备份。

## 范围边界
### 包含
- 统一管理后台 Web UI。
- 本机文件系统与 Codex 状态的读写集成。
- 角色、会话、项目、知识库、cron 五类核心对象。
- 总经理聊天面板和基础执行桥接。
- Qdrant、`.cron-loop/`、`~/.codex` 的兼容读取。

### 不包含
- 在本期内完全替代 Slack / Telegram 网关实现。
- 自建新的向量数据库或替换现有 Qdrant。
- 自研新的 cron 引擎替代系统 cron 与 `cron-loop`。
- 完整的多用户权限系统。

## 关键用例与异常用例
### 关键用例
- 用户在前端向总经理发送一条管理指令，总经理基于项目上下文调用 Codex 执行并返回结果。
- 用户新增一个角色模板，并把它设置为某项目的总经理。
- 用户查看某项目可复用的历史会话，并从既有 session 继续推进任务。
- 用户将某个 Qdrant collection 绑定到项目，并声明该项目可写入。
- 用户浏览某项目 `.cron-loop/` 任务并暂停、恢复、查看日志。

### 异常用例
- `~/.codex` 路径存在但 session 文件损坏：系统需标记为异常而非崩溃。
- Qdrant 6333 端口不可达：系统需给出降级状态并保留知识库配置。
- `.cron-loop/` 文件不完整：系统需做任务健康度告警和只读展示。
- Slack / Telegram 未配置凭证：渠道能力可见但不可执行，提示缺失配置。

## 非功能性要求
- 前端首屏可在本地开发环境中稳定加载。
- 核心 API 响应需以本地文件和本地服务为主，避免不必要联网依赖。
- 数据结构可审计、可导出、可人工编辑。
- 单文件尽量保持在 1500 行以内。
- 架构需支持后续增加更多渠道、更多角色模板和更多项目，不以单项目脚本方式收束。

## 假设清单与待确认问题
- 假设：当前环境允许执行本机 `codex` CLI，并可访问 `~/.codex`。
- 假设：Qdrant 对本机 `http://127.0.0.1:6333` 可达。
- 假设：本期先完成 Slack / Telegram 的渠道模型与接入抽象，不在没有现成凭证的前提下强行打通真实收发。
- 待确认：总经理默认发言语言是否统一中文；在无反馈情况下先按中文 UI / 中文提示实现。
- 待确认：是否要直接复用 OpenClaw Slack 网关还是只做桥接配置；在无反馈情况下先做可扩展桥接层。

## 风险与应对
- 风险：直接改写 `~/.codex/config.toml` 可能影响现有使用。
- 应对：角色与项目配置优先写入本项目自己的数据目录，通过运行时 override 物化，而不是强行覆盖全局配置。

- 风险：真实 Codex 运行输出结构可能较复杂。
- 应对：第一期采用 `codex exec --json` 的事件流兼容解析，保留原始事件记录。

- 风险：cron-loop 资产分散在多个项目目录。
- 应对：以“项目注册表 + 根路径扫描”方式建立索引，不迁移已有文件。
