# CCEO 前端操作手册

## 1. 使用原则
CCEO 的浏览器界面不是字段堆砌页。推荐始终遵循下面的顺序：

1. 先在总览页确认系统基础面是否健康。
2. 再进入某个具体工作面执行配置或治理动作。
3. 做完动作后，回到结果区或运行态确认是否真的生效。

不要把“保存成功”当成完成判定。真正完成，必须有可验证的运行态、时间戳、最新摘要或外部消息结果。

## 2. 总览页

### 2.1 什么时候来这里
- 每次打开 CCEO 的第一站
- 需要判断是去项目、渠道还是 Cron 页时
- 想知道当前还缺什么基础配置时

### 2.2 推荐操作顺序
1. 看角色、会话、Cron、MCP 四个指标卡。
2. 看 `Operator Guide / Current Context / Next Moves` 三张说明卡。
3. 看 `Trusted Roots`，确认目标项目路径已纳管。
4. 看 `知识库健康`，确认 Qdrant 在线。

### 2.3 完成判定
- 你能明确知道下一步应该进入哪一页。
- 你能判断当前是否存在 Slack 未接通、Cron 未装入、Qdrant 离线等硬阻塞。

## 3. 总经理页

### 3.1 适用场景
- 需要让总经理直接推进某项目
- 需要恢复已有 session 继续工作
- 需要同时利用角色、项目和旧会话上下文

### 3.2 标准流程
1. 先选择 `角色`。
2. 再选择 `项目`。
3. 如有需要，再从 `恢复会话` 里选择旧 session。
4. 在输入框中写明：
   - 目标
   - 范围
   - 验收条件
   - 是否允许修改代码 / 配置 / 文档
5. 点击 `发送给总经理`。
6. 观察右侧 `本次运行时间线`。

### 3.3 如何写有效 prompt
推荐写法：
- `为 CCEO 的 Slack 渠道补齐 socket mode 接入说明，并验证 connect/disconnect。`
- `扫描当前项目的 cron-loop 任务，找出最近失败的链路并修复。`
- `恢复上一次 session，继续把 Telegram 接入页做到可投产。`

不推荐写法：
- `你看一下这个。`
- `继续。`
- `修一下。`

### 3.4 完成判定
- 运行状态进入 `completed` 或 `failed`
- 右侧有真实事件，而不是空白
- 如果执行了真正动作，你能在其他页面或对象状态中看到变化

### 3.5 常见失败点
- 角色或项目没有选对，导致上下文跑偏
- 旧 session 选错，恢复到了不相关项目
- Prompt 没写验收条件，导致回复看似很多但没有可核验结果

## 4. 角色页

### 4.1 角色页负责什么
角色负责定义：
- 人格
- 模型
- 指令
- 工具 / MCP / 技能边界
- 渠道中的身份和发言规则

角色不负责：
- 保存具体历史对话
- 绑定某一个固定 session

### 4.2 新建角色步骤
1. 点击 `新建角色`。
2. 填写 `名称` 和 `描述`。
3. 选择 `模型 / Reasoning / Verbosity`。
4. 决定是否启用 `项目文档`。
5. 决定是否 `替换内建指令`。
6. 写 `System Prompt`。
7. 写 `Developer Instructions`。
8. 选择技能、MCP 和工具。
9. 填好渠道身份与回复规则。
10. 点击 `保存角色`。

### 4.3 填写建议
- `描述` 写职责，不写口号
- `System Prompt` 解决角色立场和长期规则
- `Developer Instructions` 放流程约束和执行偏好
- 技能 / MCP / 工具要按职责最小化授权

### 4.4 完成判定
- 别人看名称与描述就能知道它负责什么
- 角色被项目页选为经理或参与角色时，不需要再猜边界

## 5. 会话页

### 5.1 什么时候来这里
- 需要恢复旧会话
- 需要整理大量历史 session
- 需要给旧 session 重新补齐归属

### 5.2 标准流程
1. 查看 session 的 `cwd` 和更新时间。
2. 判断它是否属于当前项目。
3. 选择项目和角色。
4. 写备注，说明这段会话推进到了哪里。
5. 点击 `保存`。
6. 回总经理页做 `resume`。

### 5.3 备注建议
推荐：
- `已完成 Slack socket mode 字段设计，待真实 token 联调`
- `已修复 cron runner PATH，待首轮真实执行`

不推荐：
- `重要`
- `后面再看`

## 6. 知识库页

### 6.1 标准流程
1. 先看右下角的 Qdrant 状态。
2. 新建或选择知识库。
3. 填 `名称 / URL / Collection / 描述`。
4. 如果不允许项目写入，勾选 `只读`。
5. 点击 `保存知识库`。
6. 再去项目页绑定可读 / 可写权限。

### 6.2 完成判定
- URL 可访问
- Collection 不是空值
- 项目页能正确把它选入可读或可写列表

### 6.3 常见失败点
- 只保存了知识库名，没有 collection
- Qdrant 离线时仍然误以为配置完成

## 7. 项目页

### 7.1 项目页负责什么
项目是 CCEO 的治理边界，决定：
- 默认经理角色是谁
- 哪些角色参与
- 能读哪些知识库
- 能写哪些知识库
- 接了哪些渠道
- 渠道消息应该落到哪个房间和别名

### 7.2 新建项目步骤
1. 点击 `新建项目`。
2. 填 `名称`。
3. 填真实 `路径`。
4. 选择 `经理角色`。
5. 选择 `参与角色`。
6. 选择 `可读知识库`。
7. 选择 `可写知识库`。
8. 绑定渠道。
9. 为每条渠道绑定补齐 `目标房间 / 频道` 与 `项目别名`。
10. 点击 `保存项目`。

### 7.3 完成判定
- 路径正确
- 经理角色不是空值
- 绑定渠道不是只勾选不填 room
- 可写知识库与真实 collection 一致

### 7.4 常见失败点
- 只选了渠道，但没有填 room / alias
- 经理角色漏填，导致总经理运行没有默认负责人

## 8. Cron 页

### 8.1 推荐使用顺序
1. 先选项目。
2. 先对当前项目唯一的 singleton loop 执行 `validate`。
3. 阅读右侧 `动作解读`。
4. 再决定要不要 `stop / start / paths / destroy`。

### 8.2 各按钮的意义
- `validate`
  - 检查 `.cron-loop/` 结构是否完整
  - 适合第一次排查或改动后复查
- `stop`
  - 在大改动前暂停循环任务
- `start`
  - 手工改完并验证通过后恢复任务
- `destroy`
  - 彻底移除项目的 `.cron-loop/` 目录和 crontab 条目
- `paths`
  - 快速定位 prompt、runner、state、latest、lock 等文件

### 8.3 完成判定
- `validate` 返回 ok
- `latest.md` 会持续刷新
- `state.json` 不是长期停在老时间

### 8.4 CCEO 自我管理任务
当前项目自我管理任务：
- loop: `CCEO` 项目唯一 singleton loop
- schedule: `17 */4 * * *`
- timeout: `12600`

设计要求：
- 每轮先跑严格前端审计
- 门禁通过后再跑 Playwright 全盘巡检
- 同步更新文档、账本和 latest 摘要

## 9. 渠道页

### 9.1 总流程
渠道接入统一按下面顺序：

1. 选择渠道类型
2. 填当前模式所需字段
3. 保存渠道
4. 校验配置
5. Dry Run
6. 真实发送测试
7. 如果是 Slack socket mode，再执行 Connect Slack
8. 观察最近一次运行态

### 9.2 Slack 接入

#### Socket Mode
适用：
- 想让总经理常驻接入 Slack
- 不想依赖公网 webhook

步骤：
1. 类型选 `slack`
2. `Slack Mode` 选 `socket`
3. 填 `Slack Bot Token`
4. 填 `Slack App Token`
5. 按需设置 `默认项目`、`默认 Slack Target`
6. 决定是否要求频道消息必须 `@bot`
7. 点击 `保存渠道`
8. 点击 `校验配置`
9. 点击 `Dry Run`
10. 点击 `真实发送测试`
11. 点击 `Connect Slack`
12. 查看最近一次运行态中的：
    - connectionState
    - lastConnectionSummary
    - lastInboundAt
    - lastRoutedProjectId
    - lastThreadId

完成判定：
- 校验通过
- 真实发送成功
- connectionState 进入 `connected`
- 外部 Slack 中能看到测试消息

#### Webhook
适用：
- 只需要最小发消息验证

步骤：
1. `Slack Mode` 选 `webhook`
2. 填 `Slack Webhook URL`
3. 保存
4. 校验
5. Dry Run
6. 真实发送测试

注意：
- webhook 模式不负责总经理消息回流
- 一般不需要 `Connect Slack`

#### HTTP
适用：
- 你明确有 Events API 回调入口

额外字段：
- `Slack Bot Token`
- `Slack Signing Secret`
- `Slack Webhook Path`

### 9.3 Telegram 接入
步骤：
1. 类型选 `telegram`
2. 填 `Telegram Bot Token`
3. 填 `Telegram Chat ID`
4. 如无特殊代理需求，保留默认 `Telegram API Base`
5. 保存
6. 校验
7. Dry Run
8. 真实发送测试

完成判定：
- 前端最近一次投递时间更新
- 目标聊天内能看到测试消息

### 9.4 常见失败点
- token 是占位值不是实际值
- chat id 填错对象
- Slack 频道里没有把 bot 拉进去
- 只保存了配置，没有继续做校验和 live 测试

## 10. 本项目的日常巡检建议
建议每天至少检查一次：

1. 总览页下一步建议是否出现新的阻塞项
2. Cron 页里当前项目 singleton loop 的 latest 是否持续刷新
3. 渠道页的最近一次运行态是否异常
4. 文档目录中的功能清单与操作手册是否和当前界面一致

## 11. 何时判定“真的完成”
下面三个条件至少要满足两个：

- 浏览器前端能看懂，不需要猜
- 对应运行态或外部系统有可验证变化
- 文档和账本已经同步更新
