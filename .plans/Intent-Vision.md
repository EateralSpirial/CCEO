# Intent Vision

## 本次执行意图与愿景（有序号）
1. Vision 1
- 原始意图：按照 openclaw 的模式，为 codex 做一个类似性质的综合管理系统，由一个全权总经理统一管理本机上的 Codex 项目、人格、记忆、技能、MCP、工具和其他功能。
- 目标愿景：在 `workspace/` 内落地一个可运行的 Codex 管理台，拥有统一前端与总经理对话入口，并能围绕本机真实 Codex 能力执行管理操作。
- 达成判据：浏览器中可以打开管理台；管理台至少能展示和维护角色、记忆/会话、知识库、项目、cron 任务五类核心对象，并具备总经理聊天面板。
- 禁止跑偏点：不能只写概念文档或静态页面；必须接上真实的本机文件系统状态与 Codex/cron/Qdrant 运行边界。

2. Vision 2
- 原始意图：Codex 角色管理要能管理人格、偏好、system prompt、默认 system prompt 是否启用、工具集、MCP、全局/项目作用域、接入聊天工作应用时的名字与说话规则等；角色与记忆/session 解耦。
- 目标愿景：角色系统以“人格与工具集配置”为主轴，允许角色在不同项目和不同会话之间复用，并能映射到 Codex 的 profile、instructions、developer instructions、MCP、技能和工具可用性。
- 达成判据：系统中能持久化角色定义，清晰区分角色、会话、记忆；并能生成可供 `codex exec` / `codex mcp-server` 使用的运行配置。
- 禁止跑偏点：不能把角色和历史对话绑定死，也不能把“记忆”偷换成“system prompt 文案”。

3. Vision 3
- 原始意图：记忆管理、历史对话管理与归档要独立；一个历史对话不绑定单一角色，可在角色之间切换使用。
- 目标愿景：系统能够读取并管理本机 Codex 的 session/history 资产，并在项目和角色维度上做索引、标注、归档和恢复。
- 达成判据：前端中可浏览本机 Codex 会话清单、查看归属、挂接到项目，并支持基于既有 session 发起新的管理动作。
- 禁止跑偏点：不能自造一套完全脱离 `~/.codex/sessions` 的假会话体系。

4. Vision 4
- 原始意图：知识库要可管理；本机已在 `6333` 端口具备 Qdrant，可以使用；cron 任务要与现有 `cron-loop` 兼容，并可统一管理。
- 目标愿景：管理台直接纳管 Qdrant 知识库元数据与 `.cron-loop/` 平铺任务资产，形成项目维度的知识源与自动化控制台。
- 达成判据：前端能登记知识库、关联项目、显示 Qdrant 连接状态；能扫描项目内 `.cron-loop/` 目录并展示任务、状态、日志、暂停/恢复入口。
- 禁止跑偏点：不能另起一套与现有 `cron-loop` 不兼容的新任务规范，不能把 Qdrant 只当成占位字符串。

5. Vision 5
- 原始意图：总经理要有统一对话界面，并且可以接入 Telegram 或 Slack；本机 openclaw 已实现接入 Slack。
- 目标愿景：总经理拥有统一 Web 聊天入口，并为 Slack / Telegram 预留可落地的接入抽象，使后续可以把总经理投递到聊天工作面。
- 达成判据：第一期内交付统一聊天界面和渠道配置模型；后续期在不推翻架构的前提下接入 Slack / Telegram 投递与回流。
- 禁止跑偏点：不能把渠道接入写死在前端，也不能做成无法扩展到多个项目和多个角色的单通道脚本。

6. Vision 6
- 原始意图：通过 `cron-loop` 持续推进和迭代本项目，目标是前端人类可读、说明详细，灵活而且强大。
- 目标愿景：CCEO 自身拥有项目内的持续迭代回路，能够稳定推动前端说明层、操作引导、自动化治理与结构可维护性不断提升，而不是每次都靠临时人工想起再改。
- 达成判据：项目根目录存在可审计的 `.cron-loop/` 平铺任务；前端关键页面具有面向人类的说明与下一步指引；每轮迭代都有台账、状态、最新结论和验证记录。
- 禁止跑偏点：不能把持续迭代退化成“只写总结不做改动”；不能继续用原始 JSON、技术字段堆砌页面来要求用户自己理解系统。
